# MorseFold-AI Optimization Log

## 第一次优化

### 优化标记

- `Optimization #1`
- 时间：`2026-03-22`
- 范围：`encoder.py`
- 目标：在不修改解码格式、不破坏可逆性的前提下，提高简化编码的压缩收益

### 一、原始程序结构

当前程序的核心链路如下：

```text
plain text
  -> main.py:text_to_morse()
  -> encoder.py:text_to_morseSimplify()
  -> simplified code
  -> decoder.py:simplified_to_morse()
  -> decoder.py:simplified_to_text()
```

其中：

- `main.py` 负责普通文本到标准摩斯码的映射
- `encoder.py` 负责标准摩斯码到简化编码的压缩
- `decoder.py` 负责从简化编码恢复标准摩斯码和普通文本

### 二、优化前的编码逻辑

优化前，`encoder.py` 的主要策略是：

1. 在每个单词内部，把连续且长度相同的摩斯码划为一组
2. 对这一整组只做一次压缩尝试
3. 根据组内点划数量的偏向，启发式选择 `RS` 方向
4. 如果压缩后不比原始组更短，则整组回退为原始表示

示意如下：

```text
same-length run
  -> encode once
  -> shorter ? keep encoded : keep raw
```

### 三、优化前存在的问题

第一次阅读算法后，确认主要问题有两个：

1. `RS` 的方向选择不是按真实编码长度决定，而是按启发式规则决定
2. 对一个较长的同长度连续组，只尝试“整组一次编码”，不会继续搜索更优拆分

这会导致两类损失：

- 某些组虽然适合压缩，但选错 `RS` 方向后没有达到最短
- 某些大组整体压缩收益一般，但拆成两个或多个子段后会更短，旧算法无法发现

### 四、第一次优化方案

本次优化只改编码器，不改解码格式，核心思想是：

```text
对每个同长度 run，搜索最短可逆表示
```

方案拆成两层。

#### 1. `RS` 方向最优选择

对于一个候选编码段，不再先根据点划比例推断 `RS`，而是：

1. 分别构造 `RS = <length>-` 和 `RS = <length>.`
2. 分别生成两种情况下的 `ID` 串
3. 直接比较两种编码串长度
4. 选更短的那一个

这样做的好处是：`RS` 的方向由真实编码成本决定，而不是由启发式代理指标决定。

#### 2. 同长度 run 的动态规划最短分段

对于单词中的每个同长度连续段，使用动态规划搜索：

- 保留当前位置的原始码
- 或者把当前位置到某个终点的子段压缩成一个编码段

然后比较所有方案的总长度，选择最短结果。

抽象结构如下：

```text
same-length run
  -> try raw single code
  -> try encoded segment [i:j]
  -> DP choose shortest combination
```

这意味着一个较长 run 最终可以被表示为：

- 全部原样输出
- 一个压缩段
- 多个压缩段
- 压缩段和原始码混合输出

### 五、实现结构变化

第一次优化主要落在 `encoder.py` 中，新增了以下结构：

#### `build_position_ids(code)`

作用：

- 预先提取某个摩斯码中 `.` 和 `-` 的位置串
- 供两种 `RS` 方向复用

#### `best_encoded_segment(ids_for_dash, ids_for_dot, length)`

作用：

- 分别构造 `RS=-` 和 `RS=.` 的编码结果
- 直接返回长度更短的编码串

#### `optimize_same_length_run(codes)`

作用：

- 对一个同长度连续 run 做最短表示搜索
- 使用动态规划决定是保留原始码，还是压缩某个子段
- 最终输出该 run 的最短段列表

### 六、优化后的程序结构

优化后编码器的逻辑可概括为：

```text
word
  -> split into same-length runs
  -> for each run:
       compute reusable ids
       search best raw/encoded segmentation by DP
  -> join segments with |
```

整个系统链路保持不变：

```text
text -> morse -> simplified -> morse -> text
```

也就是说，本次优化是编码策略优化，不是协议格式重构。

### 七、实例说明

下面给出一个实际例子，说明第一次优化如何影响输出。

输入文本：

```text
MORSE CODE
```

标准摩斯码：

```text
-- --- .-. ... . / -.-. --- -.. .
```

旧版简化输出：

```text
--|\-\123%3-|./-.-.|---|-..|.
```

新版简化输出：

```text
--|---|-\%3.|./-.-.|\23%3-|.
```

长度对比：

- 旧版长度：`29`
- 新版长度：`28`

这个例子反映了两点。

第一，旧版会把 `--- .-. ...` 作为一个长度为 `3` 的连续组整体压缩，并按启发式选择 `RS = 3-`，得到 `\-\123%3-`。  
新版会直接比较两种方向的真实编码成本，发现把其中的 `---` 保留原样、把后两个码压成 `-\%3.` 更短，因此输出为：

```text
---|-\%3.
```

第二，在 `CODE` 对应的第二个单词中，旧版没有找到更短的整体结构；新版则会在同长度 run 内继续搜索更优子段，把 `--- -..` 压成：

```text
\23%3-
```

这说明第一次优化的价值不只是“是否压缩”，而是：

```text
允许一个同长度 run 内部出现更细粒度的最短组合
```

也就是：

- 某些码保留原样
- 某些连续子段单独压缩
- 同时为每个压缩子段重新选择最优 `RS`

#### 单词级纯示例

下面给出一个只有单词、没有跨词边界的例子，更适合单独展示分段优化。

输入文本：

```text
0123456789
```

标准摩斯码：

```text
----- .---- ..--- ...-- ....- ..... -.... --... ---.. ----.
```

旧版简化输出：

```text
12345\2345\345\45\5\\1\12\123\1234%5.
```

新版简化输出：

```text
\1\12%5-|45\5\\1\12%5.|45\5%5-
```

长度对比：

- 旧版长度：`37`
- 新版长度：`30`

这个例子里，所有字符的摩斯码长度都是 `5`，因此它们构成一个很长的同长度 run。

旧版策略会把整个 run 一次性压成一个编码段：

```text
entire run -> one encoded segment
```

新版策略则会把这个 run 拆成多个更短的编码子段：

```text
\1\12%5- | 45\5\\1\12%5. | 45\5%5-
```

这说明当 run 很长时，整组只压一次并不一定是最优的。  
第一次优化后的动态规划可以为不同子段分别选择：

- 是否压缩
- 用哪个 `RS` 方向
- 在哪里切分

因此在这种“长同长度序列”场景下，收益会比旧版明显更高。

## 第二次优化

### 优化标记

- `Optimization #2`
- 时间：`2026-03-22`
- 范围：`encoder.py`
- 目标：在保持第一次优化压缩收益和输出格式不变的前提下，降低编码器的重复计算成本

### 一、第二次优化的背景

第一次优化把编码策略从“整组一次启发式压缩”提升为“同长度 run 的最短表示搜索”，压缩率显著变好，但也引入了额外开销：

- 每次调用 `text_to_morseSimplify()` 都会重新构造内部辅助函数
- 同一个摩斯码会重复做“规则交替判断”和“位置串提取”
- 动态规划在枚举 `[start:end]` 子段时，会重复切片和重复拼接字符串

在当前数据集上，这些重复工作比较明显。统计发现：

- 总 `run` 数：`483997`
- 去重后 `run` 数：`11040`

这说明大量相同结构会在不同样本中反复出现，因此缓存和增量构造有很高的优化价值。

### 二、第二次优化方案

第二次优化不改变协议，也不改变第一次优化得到的最短分段策略，只优化求解过程本身。

核心思路：

```text
保持最优解搜索不变，把重复计算变成可复用结果
```

方案分成两部分。

#### 1. 模块级缓存

把以下纯函数移到模块级，并使用 `lru_cache` 缓存：

- `is_regular(code)`
- `build_position_ids(code)`
- `optimize_same_length_run(codes)`

这样做的目的分别是：

- 相同摩斯码不再重复判断是否为规则交替结构
- 相同摩斯码不再重复提取点位/划位位置串
- 相同的同长度 run 在不同样本中只需要求解一次

#### 2. 子段编码的增量构造

第一次优化中，动态规划在尝试每个 `[start:end]` 子段时，采用的是：

```text
slice ids -> join -> append %RS
```

第二次优化改成：

```text
extend previous segment -> append next id -> compare
```

也就是在固定 `start` 的情况下，随着 `end` 向右扩展，直接复用上一步已经构造好的 `dash_segment` 和 `dot_segment`，避免每次都重新切片和重新拼接。

### 三、实现结构变化

第二次优化后的 `encoder.py` 结构如下：

#### `is_regular(code)`

- 提升为模块级函数
- 使用 `lru_cache`
- 负责规则交替码判断

#### `build_position_ids(code)`

- 提升为模块级函数
- 使用 `lru_cache`
- 对普通码返回 `.` 和 `-` 的位置串
- 对规则交替码直接返回相同标记，避免重复分支判断

#### `pick_shorter_encoding(...)`

- 负责在 `RS=-` 与 `RS=.` 两种候选中选更短表示
- 保留第一次优化的“按真实输出长度比较”的原则

#### `optimize_same_length_run(codes)`

- 提升为模块级函数
- 使用 `lru_cache`
- 仍然执行动态规划最短分段
- 但候选子段改为增量构造，不再每次对子列表做切片拼接

### 四、第二次优化后的逻辑特征

第二次优化没有改变“找什么”，只改变了“怎么找”。

也就是说，算法目标仍然是：

```text
对每个同长度 run 找最短可逆表示
```

变化的是求解方式：

- 以前：重复计算后再比较
- 现在：复用中间结果后再比较

因此第二次优化属于：

```text
性能型算法优化
```

而不是新的压缩协议设计。

## 第三次优化

### 优化标记

- `Optimization #3`
- 时间：`2026-03-22`
- 范围：`encoder.py`
- 目标：继续提升压缩率，补齐第一次和第二次优化后仍未参与搜索的短长度 run

### 一、第三次优化的背景

在第二次优化完成后，编码器已经具备两个重要特征：

- 可以对同长度 run 做最短表示搜索
- 这套搜索过程已经被缓存和增量构造优化，计算成本明显下降

但当前实现仍保留了一个旧限制：

```text
length <= 2 的 run 直接保持原样
```

这个限制来自最初版本的经验判断，目的是避免短码因为元信息开销而“压缩反而变长”。  
问题在于，第二次优化后的编码器已经具备逐段比较真实输出长度的能力，所以这个硬限制实际上变成了一个压缩盲区。

换句话说，正确的策略不应该是：

```text
短码不压缩
```

而应该是：

```text
短码也参与搜索，但只有在更短时才压缩
```

### 二、第三次优化方案

第三次优化的核心改动非常集中：

```text
移除 length <= 2 的提前跳过逻辑
```

具体来说，`optimize_same_length_run(codes)` 不再在 `code_length <= 2` 时直接返回原始码，而是让长度 `1` 和 `2` 的 run 也进入与其他长度完全相同的最短表示搜索流程。

这意味着：

- 长度 `1` 的 run 可以被评估是否存在更短的编码段
- 长度 `2` 的 run 也可以被分段、比较、回退
- 如果编码后不更短，DP 仍然会自动回退到原始 Morse

因此第三次优化的本质不是“强制压缩短码”，而是：

```text
让短码也获得一次被证明可压缩的机会
```

### 三、实现结构变化

这次实际改动很小，主要发生在 `encoder.py`：

- 删除 `optimize_same_length_run(codes)` 中对 `code_length <= 2` 的提前返回
- 更新函数文档，明确所有同长度 run 都可以参与最短表示搜索
- 在文档中加入 `Optimization #3` 标记

因为第二次优化已经把动态规划和缓存结构整理好了，所以第三次优化不需要新增新的解码协议，也不需要修改 `decoder.py`。

## 第四次优化

### 优化标记

- `Optimization #4`
- 时间：`2026-03-22`
- 范围：`encoder.py`、`decoder.py`
- 目标：降低编码段元信息开销，在保持可逆性的前提下进一步显著提升压缩率

### 一、第四次优化的背景

前三次优化主要集中在：

- 更好地搜索“哪些 run 应该被压缩”
- 更快地完成这类最短表示搜索

到第三次优化为止，编码段本体已经比较紧凑，但每个编码段仍然保留一个固定格式的 `RS` 后缀：

```text
%<length><tail>
```

例如：

- `%3-`
- `%4.`
- `%2.`

这个设计的问题很直接：

- 每个编码段都要额外支付 3 个字符的元信息成本
- 当前项目的 Morse 码长度范围其实非常小，只有 `1` 到 `7`
- `RS` 的可能组合总共只有 `14` 种，因此完全可以用更短的固定标记来表达

也就是说，当前的元信息并不是“信息量大”，而是“表示形式冗长”。

### 二、第四次优化方案

第四次优化的核心思路是：

```text
把 RS 后缀从 3 字符压缩成 1 个保留标记
```

具体做法：

- 为 `1-`、`1.`、`2-`、`2.`、...、`7-`、`7.` 建立固定单字符映射
- 编码器输出时，不再追加 `%<length><tail>`，而是直接追加对应单字符 token
- 解码器在解析时，同时支持：
  - 旧格式 `%<length><tail>`
  - 新格式单字符 token

这次优化的本质不是改变压缩决策逻辑，而是压缩编码段本身的协议开销。

### 三、RS 单字符映射

第四次优化引入如下映射：

| RS | Token |
| --- | --- |
| `1-` | `A` |
| `1.` | `B` |
| `2-` | `C` |
| `2.` | `D` |
| `3-` | `E` |
| `3.` | `F` |
| `4-` | `G` |
| `4.` | `H` |
| `5-` | `I` |
| `5.` | `J` |
| `6-` | `K` |
| `6.` | `L` |
| `7-` | `M` |
| `7.` | `N` |

之所以可以这样设计，是因为：

- 简化编码中的原始 Morse 片段只会使用 `.` 和 `-`
- 位置标识只会使用数字
- 结构字符固定为 `\`、`|`、`/`
- 大写字母此前没有被用作协议控制符，因此适合作为保留 token

### 四、实现结构变化

第四次优化主要引入以下变化。

#### `encoder.py`

- 新增 `RS_TOKEN_TO_TEXT`
- 新增 `RS_TEXT_TO_TOKEN`
- 将原先的：

```text
%<length><tail>
```

替换为：

```text
<token>
```

例如：

- 旧：`2\2%4.`
- 新：`2\2H`

#### `decoder.py`

- 新增对单字符 token 的识别逻辑
- 仍保留对旧 `%<length><tail>` 格式的解析能力

这意味着第四次优化是：

```text
新编码格式 + 旧格式兼容解码
```

而不是破坏性协议变更。

### 五、实例说明

输入文本：


标准摩斯码：

```text

```

第三次优化输出：

```text

```

第四次优化输出：

第四次优化后的解码器支持两种输入：

```text

```

它们都会被正确恢复为相同的 Morse 序列。
